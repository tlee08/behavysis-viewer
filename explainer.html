1. Mental Models for Data Scientists

React Hooks → Data Flow Analogies

In Python data science, you write procedural scripts: load → transform → plot. React inverts this: you describe what the UI should look like given the current state, and React handles the rest. Hooks are the mechanism that connects state to rendering.

Hook What it does Python analogue
useState(initial) Creates a reactive variable. When its setter is called, the component re-renders. Like an ipywidgets.interact slider — change the value, and everything that depends on it updates automatically. Or like st.session_state in Streamlit.
useEffect(fn, [deps]) Runs fn whenever any value in [deps] changes. If [deps] is [], runs once on mount. The returned function cleans up on unmount. Like an Excel cell formula that auto-recalculates when its referenced cells change. Or a Jupyter cell you re-run manually — but automated.
useCallback(fn, [deps]) Memoizes a function so it doesn't get recreated on every render (only when deps change). Like functools.lru_cache for function identity — prevents unnecessary recomputation in child components.
useRef(initial) Mutable box that persists across renders without triggering re-renders. Like a module-level variable or a global — a value you can mutate without triggering side effects.
Concrete example from this repo (useExperimentIO.ts:7-20):

const [videoUrl, setVideoUrl] = useState<string | null>(null) — one reactive variable controls whether a video player renders.
useEffect(() => { return () => { revokeObjectURL(...) } }, []) — runs once on mount; the returned cleanup revokes the blob URL when the component unmounts.
const blobUrlRef = useRef<string | null>(null) — stores the current blob URL without causing re-renders every time it changes.

Zustand Store → Global State Dictionary / DataFrame

In Python, you might keep a global dict or pandas.DataFrame that multiple functions read and mutate. Zustand is that concept, formalized.

Concept Python (DS stack) Zustand (this app)
Store definition state = {'bouts': [], 'currentFrame': 0} create<AppState>((set) => ({ bouts: [], currentFrame: 0, ... }))
Read a value state['bouts'] useStore(s => s.bouts) in a component, or useStore.getState().bouts outside one
Mutate a value state['bouts'][i]['actual'] = 1 updateBoutActual(id, 1) — an action defined in the store that calls set(...)
React to changes Manually re-run cells / callbacks Automatic: any component using useStore(selector) re-renders when the selected slice changes
Key insight: Unlike React Context, Zustand has no Provider wrapper. Import the store hook anywhere and it just works. You can also call useStore.getState() or useStore.subscribe(callback) outside React components — no hook needed. This is how useKeyboardShortcuts.ts reads store state from a vanilla addEventListener handler.

TypeScript Types → Schema Validation for Code

Think of TypeScript types as running df.dtypes or schema validation before your code executes.

Python (runtime check) TypeScript (compile-time check)
assert isinstance(row['id'], int) interface Bout { id: number } — caught at build time, before you ever run
pd.DataFrame.astype({'start': 'int64'}) start: number — the compiler rejects bout.start = "hello"
pyarrow.schema([pa.field('actual', pa.int8())]) actual: ActualValue where type ActualValue = -1 | 0 | 1 — a union literal type
Concrete example from this repo (types.ts:8):

export type ActualValue = -1 | 0 | 1

This is a literal union type. The compiler guarantees no value other than -1, 0, or 1 can ever be assigned to an actual field. The clampActual function in bouts.ts:57-61 is a runtime safety net, but the type system catches most mistakes at build time.

Why three layers? The main process has full Node.js access (filesystem, OS dialogs) but renders no UI. The renderer process is a Chromium browser — it renders React but is sandboxed; it cannot touch the filesystem directly. The preload bridge is a narrow, typed API surface connecting the two. This is a security pattern: the renderer can only call the 6 IPC methods defined in ipc.ts.

What actually happens in the store (store.ts:99-103): 
updateBoutActual: (id, actual) =>
  set((s) => ({
    bouts: s.bouts.map((b) => (b.id === id ? { ...b, actual } : b)),
  })),
This is an immutable update: it creates a new bouts array with one element replaced, using the spread operator. React detects the reference change and re-renders. The original data is never mutated in-place — similar to df.copy() before modification.

4. The Build Pipeline

When you run pnpm dev or pnpm build, electron-vite (a wrapper around Vite) processes your source code. This is necessary because browsers (and Electron's renderer) cannot run TypeScript, JSX, or bare import statements natively.

What Vite does, step by step

Step What happens Python analogy
1. Transpile TS → JS (strips types)
JSX → React.createElement() calls 		Like Cython compiling .pyx → .c, or mypy stripping type annotations before execution
2. Resolve imports Converts bare specifiers (import 'react') to actual file paths in node_modules Like Python's sys.path resolution — finding which site-packages folder contains the module
3. Bundle Combines hundreds of .ts/.tsx files into a few .js chunks Like PyInstaller packing all .py files + dependencies into a single executable. Or pip's --target flattening into one folder
4. WASM handling vite-plugin-wasm loads .wasm files; topLevelAwait plugin enables async wasm initialization Like loading a .pyd or .so native extension in Python
5. Dev server (HMR) Watches files, hot-replaces changed modules in the browser without full page reload Like Flask's debug mode auto-reload, but surgically replacing only the changed module instead of restarting the whole process
6. Output Writes out/main/index.js, out/preload/index.js, out/renderer/ (HTML + JS + CSS) Like python setup.py build producing a dist/ folder

This project's Vite config (simplified)

// electron.vite.config.ts
export default defineConfig({
  main:    { plugins: [externalizeDepsPlugin()] },            // Node.js — keep deps external
  preload: { plugins: [externalizeDepsPlugin()] },            // Bridge — same treatment
  renderer: {
    plugins: [react(), wasm(), topLevelAwait()],              // Browser — React + WASM
    optimizeDeps: { exclude: ['parquet-wasm'] },              // Don't pre-bundle parquet-wasm
  },
})
Why exclude: ['parquet-wasm']? parquet-wasm contains a WebAssembly binary. Vite's dependency optimizer would try to pre-bundle it, which breaks the WASM initialization. Excluding it tells Vite to leave it as-is and let the wasm() plugin handle it at runtime. This is the same reason you'd use --no-binary with pip for packages that need to compile from source.

5. Code Tour: The 3 Most Important Files

src/renderer/src/hooks/useExperimentIO.ts
The file I/O orchestrator. This custom hook combines useState, useCallback, useRef, and useEffect to manage the entire open/save workflow — from picking a config file through parsing parquet with WASM to loading the Zustand store. It also handles video blob URL lifecycle (create on open, revoke on unmount), which is why the app no longer needs webSecurity: false.
src/renderer/src/store.ts
The single source of truth. A Zustand store holding all application state: bouts, keypoints, playback position, selection, and display toggles. Every component reads from it via useStore(selector), and every mutation goes through typed actions defined here (updateBoutActual, selectBout, panToFrame). The centerVisibleRange helper is a good example of derived state — it computes the visible window from the current frame.
src/main/ipc.ts
The cleanest file in the codebase. Six IPC handlers that do nothing but read/write files and return raw bytes to the renderer. There is zero data parsing here — the main process has no idea what a "bout" or "keypoint" is. This separation means all the logic lives in one process (the renderer), making the data flow easy to trace: disk → main (raw bytes) → preload (passthrough) → renderer (parse → store → UI).
Honorable mention: src/renderer/src/lib/parseParquet.ts
The data science heart of the app. Converts parquet bytes → WASM → Apache Arrow Table → per-frame maps → bouts (via run-length encoding in bouts.ts). Handles pandas-style 2-level and 4-level MultiIndex tuple column names (e.g., "('attack', 'actual')") with regex parsing. The save path does the reverse — reconstructs a valid parquet with the original schema intact.

6. Self-Check

Test your understanding. Click each question to reveal the answer.

Where does parquet parsing happen — in the main process or the renderer?

The renderer. The main process (ipc.ts) only reads raw file bytes via readFileSync and returns them as Uint8Array. parseParquet.ts in the renderer then calls parquet-wasm (a WebAssembly library) to parse those bytes into an Apache Arrow table. This design keeps all data logic in one process.

If I want to add a new keyboard shortcut, which file do I edit?

useKeyboardShortcuts.ts. It subscribes to the Zustand store directly (not via the React hook) so it runs in a vanilla keydown event listener. Add a new case in the switch statement, call an action from useStore.getState(), and you're done. No component changes needed — the store actions update state, and subscribed components re-render automatically.

Why do we need pnpm-workspace.yaml with allowBuilds?

Packages like electron, esbuild, and @swc/core ship native binaries that must be compiled or downloaded for your platform during pnpm install. The allowBuilds list tells pnpm "yes, these packages are allowed to run their postinstall scripts." Without it, pnpm would block them and the native modules would be missing — Electron wouldn't launch, and Vite's internal bundler (esbuild) would fail.
